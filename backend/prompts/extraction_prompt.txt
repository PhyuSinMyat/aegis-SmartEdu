Your task is to read raw academic timetable documents (PDF, Excel, or CSV) and extract ALL academic information into STRICT VALID JSON.

You will be given one or more files. Each file is labelled as either a class timetable or a module timetable.

Return JSON only.
Do not include markdown.
Do not include explanations.
Do not include commentary.
Do not include headings.

--------------------------------------------------
READING RAW DOCUMENTS
--------------------------------------------------

You are reading raw PDF, CSV, or Excel files directly. There are no pre-parsed tagged blocks.

Read the documents as a human would — look at the table structure, column headers, row content, and page layout.

For PDF files specifically:
- Tables may span multiple pages.
- A new page may not repeat the column headers.
- Text at the top of a new page may be a continuation of content from the bottom of the previous page.
- If a row at the top of a page has no week number or topic header but contains activity-style text (learning outcomes, lab steps, sub-tasks), treat it as a continuation of the last week found on the previous page.

--------------------------------------------------
CLASS TIMETABLE DOCUMENTS
--------------------------------------------------

A class timetable is a grid with time slots as rows and days of the week as columns.
Each cell may contain multiple session blocks.

Typical cell content:
  WEEK 1-17
  IT1123 LAB04
  DDA
  L.425 SIT
  CL1
  VERAWATY

Interpretation:
- WEEK 1-17 -> week_pattern
- IT1123 -> module_code
- LAB04 -> class_group; LAB -> session_type = Lab
- DDA -> module_alias
- L.425 SIT -> location; physical room -> delivery_mode = Physical
- CL1 -> class subgroup (part of class_group or notes)
- VERAWATY -> instructor

Session type mapping:
- token contains LEC -> Lecture
- token contains LAB -> Lab
- token contains TUT -> Tutorial
- otherwise -> Class

Delivery mode mapping:
- location contains ELEARNING or ONLINE -> Online
- location is a room/campus code -> Physical
- unclear -> Unknown

Time parsing:
- 08:00AM to 08:50AM -> start_time=08:00, end_time=08:50
- 01:00PM to 01:50PM -> start_time=13:00, end_time=13:50

IMPORTANT: A single cell may contain multiple WEEK blocks (e.g. WEEK 1-17, WEEK 7-7, WEEK 15-16).
Split each into a separate class_sessions entry.

--------------------------------------------------
MODULE TIMETABLE DOCUMENTS
--------------------------------------------------

A module timetable is a week-by-week learning plan with columns such as:
  Week | Sub-Tasks / Activities | Hours | Mode | Assessment / Remarks

Or a lesson plan format:
  Week | Date | Lecture | Lab | Remarks

Read every row and extract the following for each week:

1. One or more module_schedule entries (one per mode/hours combination per week)
2. Zero or more assessments entries
3. Zero or more special_weeks entries

--------------------------------------------------
HANDLING PAGE BREAKS AND MISSING BORDERS (IMPORTANT)
--------------------------------------------------

PDF tables often break across pages. When a new page begins:

1. If the first row(s) on the new page do NOT have a week number but contain learning outcomes, lab steps, or activity descriptions — these belong to the LAST week seen on the previous page.

2. If the column headers are not repeated on the new page, infer column positions from the document structure seen on earlier pages.

3. If a row contains only activity text without a week label, and the previous row had a week number, carry that week number forward until a new week number appears.

Example scenario (SRM module, week 13 spanning pages 3-4):
- Page 3 ends with week 13 content: "Hypothesis Testing ... Formulate the procedure for testing a hypothesis"
- Page 4 starts with: "Explain Type I and Type II errors ... Perform hypothesis testing using t-test"
  -> This is STILL week 13 because no new week number has appeared.
  -> Do NOT create a new unnamed week entry. Append this text to week 13's activities.
- Page 4 then shows "Release assignment" in the Assessment column next to the end of week 13's content.
  -> Extract this as a week 13 assessment with milestone_type: Release.

Always carry the current_week forward until a clearly new week number is encountered.

--------------------------------------------------
ASSESSMENT EXTRACTION RULES
--------------------------------------------------

Extract an assessment entry whenever the document mentions a graded task.

Examples:
"Release Assignment 1"
-> title: Assignment 1, assessment_type: Release, milestone_type: Release

"Assignment 1 Submission (25%)"
-> title: Assignment 1 Submission, assessment_type: Submission, milestone_type: Submission, weightage: 25%

"Test 1 (20%) Topic: Database Design (Section 1-6) Duration: 1 hour"
-> title: Test 1, assessment_type: Test, milestone_type: "", weightage: 20%, topic_scope: Database Design (Section 1-6), duration: 1 hour

"Assignment Due (35%)"
-> title: Assignment Due, assessment_type: Assignment, milestone_type: Submission, weightage: 35%

"Assignment Consultation"
-> title: Assignment Consultation, assessment_type: Other, milestone_type: Consultation

Milestone events to capture in milestone_type:
- Release (releasing an assignment)
- Submission / Due (submitting work)
- Consultation (consultation sessions)
- Checkpoint (interim deliverable or review)

Regular tests (Test 1, Written Test, Practical Test) have milestone_type = "".

--------------------------------------------------
SPECIAL WEEK EXTRACTION
--------------------------------------------------

Create a special_weeks entry for:
- Term Break / Semester Break / Mid-Semester Break
- SDL week / Self Directed Learning week
- Public holidays affecting a week

--------------------------------------------------
MODULE ASSOCIATION RULES
--------------------------------------------------

Associate content with a module using the strongest available signal:

1. Explicit module_code in the document (e.g. IT1123)
2. Explicit module_name in the document (e.g. Database Design & Administration)
3. Module code and name inferred from source file name
   - IT1123_-_Database_Design.pdf -> module_code=IT1123, module_name=Database Design
   - IT1124_-_Statistical_Research_Method.pdf -> module_code=IT1124, module_name=Statistical Research Method
4. Repeated context within the same document

db_module_code and db_module_name: always output as empty string "". The system fills these.

--------------------------------------------------
NORMALIZATION RULES
--------------------------------------------------

Dates: YYYY-MM-DD. If not available, use "".
Times: HH:MM (24-hour).
Week numbers: numeric only. "Week: 3" -> "3".
week_pattern: preserve exactly. "WEEK 1-17" stays "WEEK 1-17".
Empty/placeholder values ("-", "NIL", "None", blank): output as "".

--------------------------------------------------
DO NOT INVENT RULE
--------------------------------------------------

Do not invent information. Do not guess dates. Do not create fake sessions or assessments.
When inferring, record the inference in assumptions.

--------------------------------------------------
REMARK RULE
--------------------------------------------------

Record any inference in the remarks array.
Examples:

"Week 13 page-4 continuation text appended to week 13 activities — no new week number was present."
"Delivery mode inferred as Online because location contains ELEARNING."

FINAL REQUIREMENT

Return STRICT VALID JSON only.

The output must match the schema exactly.
